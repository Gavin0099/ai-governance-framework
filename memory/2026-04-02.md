## Memory Closeout Visibility

- Confirmed a real workflow gap in consuming repos: repeated code changes can leave
  `memory/` unchanged because the shared path does not make memory closeout
  decisions visible.
- Landed a first closeout-visibility slice in `session_end`:
  - detect promotion-worthy candidate signals
  - emit `memory_closeout` in summary/result artifacts
  - render human-readable closeout lines so "no memory update" is explained
- Current closeout payload now makes these questions observable:
  - was a candidate detected
  - was promotion considered
  - what decision was reached
  - why no durable write happened
- First detected candidate signals include:
  - `crash_or_runtime_failure`
  - `public_api_change`
  - `architecture_boundary_risk`
  - `proposal_risk_or_required_evidence`
  - `regression_fix`
- This slice does **not** force auto-write for major bug fixes; it closes the
  silent gap by making closeout outcomes reviewable.
