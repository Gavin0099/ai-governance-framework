# 2026-04-01

## Agent-Assisted Adoption Baseline

### What Landed
- Split beta gate semantics into two distinct tracks:
  - human self-serve onboarding
  - agent-assisted adoption
- Wrote down the semantic rule that human self-serve failure does not invalidate
  agent-assisted adoption, and vice versa.
- Added explicit agent boundary language:
  - allowed inputs
  - forbidden compensation
  - evidence boundary
- Added `docs/beta-gate/agent-run-sheet.md` so agent runs are replayable,
  challengeable, and comparable rather than one-off success stories.
- Recorded the first formal agent-assisted adoption baseline in
  `docs/beta-gate/agent-run-2026-03-31.md`.
- Added one live DBL execution artifact to that baseline via
  `runtime_hooks/core/pre_task_check.py` against
  `examples/decision-boundary/minimal-preconditions/contract.yaml`.
- Clarified `quickstart_smoke.py` output so repo-default contract mode is explicit:
  - `contract_mode=repo-default`
  - `contract_note=no explicit --contract provided; using repo-local governance defaults`

### What This Changed
- Agent-assisted adoption is now a primary evaluation surface rather than a
  side-effect of human onboarding.
- The framework now has a first auditable agent execution model:
  - replay hint
  - decision checkpoints
  - non-action log
  - inference classification
  - auditability judgment
- The first agent baseline no longer carries an unresolved `contract_path=None`
  ambiguity; the remaining weakness is drift-finding triage, not quickstart
  interpretation.

### What Is Still Not Proven
- One successful agent-assisted baseline is not yet the same thing as a stable,
  repeatable agent adoption capability across reviewers or agent types.
- Drift findings in the agent baseline were recorded as-found but not triaged;
  auditability is improved, but not yet fully closed.
- Agent-assisted adoption still needs at least one additional run or replay to
  show comparability, not just a single baseline example.

### Boundary Notes
- Human self-serve failure is acceptable for this repo direction as long as the
  agent-assisted path remains governance-safe, reviewable, and effective.
- `_integration_check.py` remains intentionally outside this milestone.
- Existing untracked audit/publication outputs remain non-source outputs and are
  not part of this change set:
  - `.governance-audit/`
  - `artifacts/reviewer-handoff/`
  - `docs/payload-audit/onboarding-2026-03-23.jsonl`

### Likely Next Moves
- Run one additional agent-assisted adoption replay or variant run to test
  whether the baseline is comparable across executions.
- Or perform a focused follow-up on drift-finding triage so the current agent
  baseline becomes cleaner as audit evidence.

## Reference Scan - `instructkr/claw-code`

### Source checked
- GitHub repo: `https://github.com/instructkr/claw-code`

### Working judgment
- `claw-code` is not a direct governance-framework peer to this repo.
- It is better understood as an execution harness / orchestration runtime
  reconstruction project, while `ai-governance-framework` is focused on
  governance runtime, evidence, verdicts, reviewer handoff, and decision
  accountability.
- Treat it as an execution-substrate reference, not as a replacement for this
  repo's governance model.

### What looks reusable
- Harness/runtime surface decomposition:
  - command inventory
  - tool inventory
  - task lifecycle
  - workspace manifest
  - parity / coverage thinking
- Manifest-first observability:
  - the idea of making runtime surfaces queryable before claiming deeper
    equivalence or maturity
- Naming separation:
  - execution harness layer vs governance runtime layer vs review/memory layer
- Multi-agent / role-oriented orchestration pressure:
  - useful prompt to formalize which roles can supply evidence, candidate
    verdicts, or final authority

### What should not be copied
- Legal / leak / clean-room backstory framing
- Research-repo maturity claims as if they were enterprise governance posture
- A "build execution first, govern later" product rhythm

### Feasibility judgment
- Feasible as a selective reference for:
  - runtime surface manifests
  - inventory-first CLI or manifest views
  - coverage / parity checkpoints for execution surfaces
- Not feasible as a full design template, because its center of gravity is
  harness/runtime equivalence, not governance correctness.
- The safest interpretation is:
  - `claw-code` helps answer "what does an agent runtime surface look like?"
  - this repo answers "how is that surface governed, audited, and made
    accountable?"

### Strategic implication
- A good next borrowing candidate would be a small runtime-surface manifest for
  this repo:
  - adapter inventory
  - tool / entrypoint inventory
  - evidence-source inventory
  - authority-boundary inventory
- This should only be done if kept clearly below the current governance runtime
  layer; it should not collapse execution harness and governance semantics back
  together.
