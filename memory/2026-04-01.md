## Reference Scan - `instructkr/claw-code`

### Source checked
- GitHub repo: `https://github.com/instructkr/claw-code`

### Working judgment
- `claw-code` is not a direct governance-framework peer to this repo.
- It is better understood as an execution harness / orchestration runtime
  reconstruction project, while `ai-governance-framework` is focused on
  governance runtime, evidence, verdicts, reviewer handoff, and decision
  accountability.
- Treat it as an execution-substrate reference, not as a replacement for this
  repo's governance model.

### What looks reusable
- Harness/runtime surface decomposition:
  - command inventory
  - tool inventory
  - task lifecycle
  - workspace manifest
  - parity / coverage thinking
- Manifest-first observability:
  - the idea of making runtime surfaces queryable before claiming deeper
    equivalence or maturity
- Naming separation:
  - execution harness layer vs governance runtime layer vs review/memory layer
- Multi-agent / role-oriented orchestration pressure:
  - useful prompt to formalize which roles can supply evidence, candidate
    verdicts, or final authority

### What should not be copied
- Legal / leak / clean-room backstory framing
- Research-repo maturity claims as if they were enterprise governance posture
- A "build execution first, govern later" product rhythm

### Feasibility judgment
- Feasible as a selective reference for:
  - runtime surface manifests
  - inventory-first CLI or manifest views
  - coverage / parity checkpoints for execution surfaces
- Not feasible as a full design template, because its center of gravity is
  harness/runtime equivalence, not governance correctness.
- The safest interpretation is:
  - `claw-code` helps answer "what does an agent runtime surface look like?"
  - this repo answers "how is that surface governed, audited, and made
    accountable?"

### Strategic implication
- A good next borrowing candidate would be a small runtime-surface manifest for
  this repo:
  - adapter inventory
  - tool / entrypoint inventory
  - evidence-source inventory
  - authority-boundary inventory
- This should only be done if kept clearly below the current governance runtime
  layer; it should not collapse execution harness and governance semantics back
  together.
- Follow-up design correction:
  - a pure passive manifest is too weak
  - the better posture is soft enforcement via manifest consistency signals
  - first useful signals:
    - unknown surface
    - orphan surface
    - evidence surface mismatch
  - these should not change verdicts in the first slice, but they should
    prevent manifest/reality drift from remaining silent

## Host-Agent Memory Gap

### What Was Clarified
- The current problem is not "memory feature missing" in a narrow sense.
- The real gap is that four layers were being treated as one chain:
  - instruction
  - repo artifact / repo memory
  - host-agent memory
  - enforcement / integration
- The framework currently has repo memory and artifact generation, but not a
  verified host-agent memory sync loop.

### More Precise Diagnosis
- `copilot-instructions.md` and similar files are guidance only:
  - they can influence behavior
  - they cannot guarantee execution
  - they cannot prove completion
- repo memory and runtime artifacts are repo state, not host memory state
- a host-agent adapter alone would not fully solve this; a sync policy is also missing

### Proposed Gap Framing
- `host-agent memory integration gap`
- `memory sync policy gap`

### Recommended Missing Pieces
- boundary clarification
- sync policy definition
- enforceable evidence loop

### Strategic Note
- Memory sync should eventually be treated as a first-class governance event,
  not as a best-effort reminder handled only by agent self-discipline.

## Consuming Repo Memory Schema Gap

### What Was Confirmed
- A consuming repo that adopts the framework can currently end up with only
  `memory/02_tech_stack.md` or `memory/02_project_facts.md`.
- This was not primarily a Copilot issue. It was an adoption/schema gap:
  - `adopt_governance.py` did not scaffold a minimum memory file set
  - `governance_drift_checker.py` did not emit any signal for partial memory schema
  - `external_project_facts_intake.py` accepted partial memory intake silently

### What Landed
- `adopt_governance.py` now scaffolds the minimum canonical memory files for new repos:
  - `memory/01_active_task.md`
  - `memory/02_tech_stack.md`
  - `memory/03_knowledge_base.md`
  - `memory/04_review_log.md`
- `governance_drift_checker.py` now emits a warning-level
  `memory_schema_complete` signal when the consuming repo is missing logical
  memory files or accepted aliases.
- `external_project_facts_intake.py` now marks intake as:
  - `memory_schema_status=partial`
  - or `memory_schema_status=complete`
  and lists `missing_logical_names`.
- `external_repo_readiness.py` now surfaces that partiality explicitly rather
  than treating a single `02_*` file as silently complete.

### Verification Notes
- `pytest tests/test_external_project_facts_intake.py tests/test_external_repo_readiness.py -q`
  passed (`18 passed`).
- `pytest` runs that rely on `tmp_path` still hit the known Windows permission
  issue on this machine; for the affected adopt/drift paths, verification was
  done via:
  - `python -m py_compile ...`
  - a manual local adopt/drift/intake run under `tests/_tmp_manual_memory_schema_check`
- Manual verification confirmed:
  - adopt now writes the full `01/02/03/04` scaffold
  - intake reports `memory_schema_status=complete` once all four logical files exist
  - drift reports `memory_schema_complete=True` when the scaffold is present

### Strategic Note
- This closes the most misleading part of the old behavior:
  the system no longer lets `02_*` act like silent proof that the full memory
  schema exists.
