# 治理 vs Prompting

> 為什麼「定規則」比「餵指令」更重要

---

## 問題的根源

大多數工程師使用 AI 的方式是 **Prompting**：

```
「你是一個資深工程師，請幫我用 C++ 實作一個...」
「記住，要遵循 SOLID 原則...」
「上次你說的方法有問題，請改成...」
```

這個模式有一個根本性的缺陷：

> **每次對話都是從零開始。**

---

## Prompting 的三大問題

### 問題一：記憶不持久

```
對話 1：你定義了架構規則 A
對話 2：AI 忘了規則 A，做了違反的事
對話 3：你重新解釋規則 A
對話 4：AI 又忘了...
```

工程師成為 AI 的「外部記憶體」，不斷重複自己。

### 問題二：規則無法被驗證

Prompt 裡說「遵循 SOLID 原則」，但：
- AI 對 SOLID 的理解和你一樣嗎？
- 什麼情況下 AI 會認為違反了原則？
- 誰來驗證 AI 確實遵守了？

模糊的指令產生模糊的結果。

### 問題三：無法跨對話維持一致性

一個功能橫跨 10 次對話完成時：
- 第 3 次對話的決策，第 8 次的 AI 還記得嗎？
- 架構邊界是否在每次對話都被守護？
- 技術債是否在不知不覺中累積？

---

## 治理框架的不同之處

| 維度 | Prompting | Governance |
|------|-----------|------------|
| **規則載體** | 對話訊息（揮發性） | 結構化文件（持久性） |
| **規則清晰度** | 模糊（「遵循最佳實踐」） | 明確（「Domain 不得呼叫 P/Invoke」） |
| **執行方式** | AI 自行解讀 | AI 有明確的檢查點和停止條件 |
| **跨對話一致性** | 無保證 | 每次對話強制載入，行為一致 |
| **優先級管理** | AI 猜測 | PLAN.md 明確定義，AI 主動對齊 |
| **停機條件** | 無定義 | HUMAN-OVERSIGHT.md 明確定義何時必須停 |

## 具體對比：加一個功能

### Prompting 方式

```
User: 幫我加個快取功能

AI: 好的！我在 UserService 裡加入了 MemoryCache...
    （花了 2 小時，加了 300 行代碼）

User: 等等，這個月我們還在 Phase A，
      不是應該先做核心功能嗎？

AI: 抱歉，我不知道你的開發階段...
    要我撤銷這些修改嗎？

User: （崩潰）
```

### Governance 方式

```
User: 幫我加個快取功能

AI: 我讀取了 PLAN.md：
    - 當前 Phase A：核心功能開發
    - 本週聚焦：登入、CRUD、API 整合
    - Phase A 禁止：不要提前優化（Phase C 才做）

    快取屬於效能優化，在「不要做」清單中。

    選項：
    A) 遵守計畫，Phase C 再處理快取
    B) 如果快取是核心需求，調整 PLAN.md

    你希望如何處理？
```

**差異**：AI 在做任何事之前，先對齊優先級。

---

## 治理文件 vs Prompt 的本質差異

**Prompt** 是對單次對話的指令：
```
「你是一個謹慎的工程師，請遵循...」
```
→ 這次對話有效，下次對話忘記。

**治理文件** 是對 AI 行為的規格書：
```
## Red Lines (any trigger → STOP)
- Implicit tech debt (no exit strategy)
- Logic leakage (Domain touching OS/I/O/UI/Time)
- Ambiguous intent ("roughly", "try to")
```
→ 每次對話強制載入，行為一致可預測。

---

## 什麼時候用 Prompting？什麼時候需要治理？

### 適合 Prompting 的場景

- 一次性查詢（「解釋這段代碼」）
- 單次任務（「幫我翻譯這段文字」）
- 探索性實驗（「試試看這個方法」）

### 需要治理框架的場景

- **長期專案**（超過 1 週的開發）
- **多 Phase 開發**（有明確的開發階段）
- **團隊協作**（需要對齊優先級和規範）
- **有架構邊界**（Domain/Infrastructure 分離）
- **需要可審計性**（決策需要可追溯）

---

## 治理框架不是在限制 AI

一個常見的誤解是：「這些規則會讓 AI 變慢、變笨。」

事實相反：

> **清晰的邊界讓 AI 在正確的範圍內更快、更準。**

沒有治理框架，AI 在猜測你的意圖；
有了治理框架，AI 在執行你已經定義好的決策。

前者的「速度」是表象，後者的工作才能累積成產品。

---

## 結語

Prompting 是戰術層面的技巧。
Governance 是戰略層面的架構。

就像軟體開發中：
- 會寫代碼 = 戰術
- 會設計架構 = 戰略

兩者都需要，但層次不同。

AI Governance Framework 讓你從「每天給 AI 指令」升級為「一次定義規則，AI 永遠遵守」。
