# Agent Instruction Surface Map

Status: design/reference note
Runtime behavior change: no
Enforcement change: no
Consumer repo change: no

## Purpose

This note maps where AI Governance instructions can reach different agent
surfaces. It exists because recent update and review failures had the same
shape: a rule existed in the framework, but the active agent surface did not
see it at execution time.

This document is a map, not a gate. It does not prove that any agent loaded,
understood, or obeyed an instruction.

## Governance Surface Expansion Gate

This note answers the three questions from `governance/AGENT.md` Section 12.

1. Is this governance-critical?

Yes. The observed failure weakens evidence accountability: update-reporting,
review, and manual-update rules can be correct in the framework while still
missing from the active agent instruction surface.

2. What existing rule cannot already cover it?

Existing protocol files define what agents should do after they are loaded.
They do not identify which delivery surface carries those rules to Codex,
Copilot, Claude, or a consuming repo's update session. This missing map led to
rules being correct but invisible.

3. What is the minimal added surface?

One reference note plus one discovery pointer from
`governance/AI_GOVERNANCE_UPDATE_PROTOCOL.md`. No runtime hook, validator,
gate, template rewrite, or consumer repo edit is added by this slice.

## Managed Block Warning

Managed instruction blocks are generated surfaces. They are useful because they
put current framework requirements into the consuming repo's active instruction
file, but they also have two risks:

- hand edits inside a managed block can be overwritten by the next refresh;
- unmanaged legacy instructions can remain outside the marker block and conflict
  with the current managed block.

When debugging update-reporting behavior, do not only check whether the new
managed block exists. Also check whether an older conflicting instruction
remains outside the managed block.

Known managed marker:

```text
<!-- governance:key=f7_update_boundary -->
```

The current F-7 block is generated from
`governance_tools/f7_full_update.py`.

## Agent-Specific Notes

### Codex

Codex usually sees repo-local `AGENTS.md` and the current task context. For AI
Governance update behavior, the most reliable repo-visible delivery path is the
consumer repo `AGENTS.md` managed block plus tool output from F-7/updater.

If Codex bypasses framework tools and manually changes a submodule pointer or
lock file, the framework cannot prove compliance. The correct status is
`manual_update` or `destructive_manual_update`, not `updated` or
`completed`.

### Copilot

Copilot-facing guidance is delivered through `.github/copilot-instructions.md`
when `governance_tools/hook_installer.py` deploys
`governance/copilot-instructions-template.md`.

This is an instruction surface, not evidence that Copilot followed the
instruction. Hook installation and instruction deployment must be reported as
placement evidence only.

### Claude

Claude may have user-level playbooks or external memory that are not part of
this repository. Those materials can guide an operator, but they are not repo
governance authority unless copied or summarized into a repo-governed surface.

When Claude review lessons are durable and cross-agent relevant, convert them
into a repo-governed note, protocol, test, or checklist instead of relying on a
private playbook alone.

## Update Reporting Rule Of Thumb

For AI Governance update tasks, the final report should be driven by tool
output that appears in the same run:

- prefer F-7/updater human output when available;
- if JSON is used, relay the `human_readable_adoption_summary` table rows and
  `final_report_requirement`;
- if no governed updater/F-7 output exists, report `manual_update` and do not
  claim completed/latest/full adoption.

## Claim Ceiling

This map does not claim:

- any agent loaded a listed surface;
- any model complied with listed instructions;
- hooks are installed on every developer machine;
- Copilot, Claude, Codex, or another agent shares the same context model;
- consumer repos are fully governed;
- instruction placement is enforcement;
- manual update paths are prevented.

It only records where repo-governed instructions are intended to be surfaced and
what each surface cannot prove.
