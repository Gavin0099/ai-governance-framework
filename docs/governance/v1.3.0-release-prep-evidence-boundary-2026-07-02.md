# v1.3.0 Release-Prep Evidence Boundary - 2026-07-02

## Problem

`PLAN.md` currently blocks v1.3.0 release-prep until two things are explicitly
scoped:

- release-surface consistency;
- consumer-side proof.

The problem is not that v1.3.0 release work should start now. The problem is
that the repository needs a reviewable definition of what evidence would be
enough before any future release-prep slice can safely begin.

Without that boundary, candidate tooling, adoption reporting, and Unreleased
CHANGELOG entries could be misread as release readiness or consumer adoption.

## Decision

Before any v1.3.0 release-prep slice starts, the repository must have both:

1. a release-surface consistency packet showing that public and reviewer-facing
   repo surfaces make the same bounded claim; and
2. a consumer-side proof packet for each capability the release intends to claim
   as consumer-facing.

If either packet is missing, stale, or ambiguous, the correct release state is:

```text
v1.3.0 release-prep: not started
release-surface consistency: not yet sufficient
consumer-side proof: not yet sufficient
claim ceiling: Unreleased candidate or report-only surface, as applicable
```

## Target Outcome

This note defines the evidence threshold for a future v1.3.0 release-prep
authorization decision.

It does not collect that evidence, run release tooling, write a release note,
publish a release, or bless a release candidate.

## Scope

In scope:

- define sufficient release-surface consistency evidence;
- define sufficient consumer-side proof evidence;
- list gaps and non-claims;
- recommend the smallest next evidence collection tranche.

Out of scope:

- version bump;
- release package generation;
- GitHub release drafting;
- tag creation;
- release publication;
- runtime hook, CI, pre-push, or gate behavior changes;
- provider prompt-cache integration;
- consumer repo writes;
- fleet rollout;
- semantic correctness or production readiness claims.

## Release-Surface Consistency Evidence Required

Before v1.3.0 release-prep can start, a release-surface consistency packet must
show all of the following.

1. Surface inventory

   The packet must name the exact release-facing and reviewer-facing files
   checked, at minimum:

   - `README.md`;
   - `CHANGELOG.md`;
   - `PLAN.md`;
   - `docs/REVIEWER_ENTRYPOINT.md`;
   - `docs/ADOPTION_MODEL.md`;
   - `docs/releases/README.md`;
   - any new `docs/releases/v1.3.0*` file, if it exists;
   - any status, badge, topic, package, or generated release artifact that the
     release text intends to cite.

2. Claim-class alignment

   Each cited capability must keep the same claim class across surfaces:

   - `Enforced` only when the exact blocking scope and evidence are named;
   - `Advisory` only when the surface reports but does not block;
   - `Observation` only when the surface produces evidence but is not a gate;
   - `Cannot claim` or `candidate-only` when implementation or proof is
     incomplete.

3. Unreleased-to-release disposition

   Every relevant `CHANGELOG.md` Unreleased entry must be classified as one of:

   - included in v1.3.0 with evidence references;
   - deferred from v1.3.0 with a clear non-claim;
   - candidate-only and not part of the release claim.

4. Stale-claim scan

   The packet must include a targeted grep or equivalent review for terms that
   would inflate the release claim, including:

   - `v1.3.0`;
   - `release-prep`;
   - `release readiness`;
   - `production ready`;
   - `governance proven`;
   - `AUTHORITY_MANIFEST`;
   - `final_report_requirement`;
   - `governance_maturity_summary`.

5. Fleet freshness boundary

   Any release claim citing fleet state must include a fresh event-driven fleet
   snapshot generated for the release decision. An older ratio or idle-period
   ratio is not sufficient.

6. Publication surface boundary

   If badges, topics, release notes, generated package artifacts, or GitHub
   release drafts are in scope, the packet must show they match the same claim
   ceiling as PLAN and CHANGELOG. If they are not in scope, the packet must say
   they are not part of the release-prep claim.

7. No contradictory authority source

   If README, release docs, reviewer docs, memory, and PLAN disagree, PLAN is
   the active claim boundary until a separate PLAN update resolves the conflict.

## Consumer-Side Proof Evidence Required

Before v1.3.0 release-prep can claim any consumer-facing capability, the
consumer-side proof packet must show all of the following for each claimed
consumer-facing capability.

1. Named consumer identity

   The packet must name the consumer repo, path, role, owner or operator, and
   pinned framework commit. The consumer role must be classified using
   `docs/ADOPTION_MODEL.md`.

2. Consumer-side execution

   Evidence must come from the consumer side, not only from this repository's
   simulation or unit tests. For example, an adoption/update claim needs the
   consumer's report path, command output, or closeout receipt that shows the
   consumer observed the result.

3. Positive path

   The evidence must show the claimed happy path completed at the consumer:

   - adoption or update report emitted the expected human summary table;
   - `final_report_requirement` was visible where the final report must relay
     it;
   - lock/check-out status was reported within the claim boundary;
   - required report path or artifact exists and is inspectable.

4. Negative or fallback path

   At least one relevant failure or boundary path must be exercised or explicitly
   ruled out for the release claim. Examples:

   - maturity summary `not_available` fallback preserves explicit
     `claim_boundary`;
   - stale or mismatched framework pin is visible as a report-only signal;
   - dirty tree or unsupported consumer role is reported without upgrading the
     claim;
   - missing or malformed `AUTHORITY_MANIFEST` is not treated as cache reuse
     proof.

5. Exact evidence references

   The packet must include commands, artifact paths, commits, and report paths.
   A prose statement that a consumer "worked" is not sufficient.

6. No semantic overreach

   The packet must state what it does not prove, including:

   - fleet generality;
   - production readiness;
   - semantic correctness;
   - non-bypassable enforcement;
   - branch protection;
   - provider prompt-cache hit/miss;
   - every consumer role.

7. AUTHORITY_MANIFEST special boundary

   If v1.3.0 release-prep intends to include the `AUTHORITY_MANIFEST` preflight
   path as more than candidate tooling, the packet must include a named real
   harness consumer and the evidence contract from the existing preflight
   decision note. Otherwise, `AUTHORITY_MANIFEST` stays Unreleased
   candidate-only and must be excluded from the release claim.

## Boundary And API Considerations

This is a documentation and planning boundary. It must not be cited as an API
contract, runtime behavior change, release gate, or consumer evidence source.

Future implementation slices may use this note to decide what to collect, but
they still need their own DONE, allowed files, validation, commit/push intent,
and non-claims.

## Failure Paths Or Risk Points

- Release inflation: Unreleased entries or candidate tooling may be read as a
  release-ready capability.
- Consumer generalization: one consumer's evidence may be overclaimed as fleet
  proof.
- Advisory-to-enforcement drift: reporting surfaces such as
  `governance_maturity_summary` may be treated as gates.
- Harness simulation overclaim: repo-side `AUTHORITY_MANIFEST` preflight tests
  may be treated as real harness behavior.
- Stale release docs: `docs/releases/README.md` still names v1.1.0 as the last
  formal release; v1.3.0 prep must not silently backfill newer main-branch work
  into a formal release.

## Evidence Plan

The smallest evidence plan is two separate read-only packets, in this order:

1. Release-surface consistency packet
   - inspect the release-facing surfaces listed above;
   - grep for stale or inflated release claims;
   - classify each Unreleased item as included, deferred, or candidate-only;
   - record whether publish surfaces are absent, deferred, or aligned.

2. Consumer-side proof packet
   - name exactly one consumer capability and one consumer repo/path;
   - classify the consumer role;
   - collect consumer-side command/artifact evidence;
   - record positive and negative or fallback evidence;
   - state non-claims and whether the evidence is enough for only that
     capability.

Do not merge these two packets unless the future DONE explicitly authorizes both
and names the consumer.
