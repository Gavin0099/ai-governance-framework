# AI Governance 文件用語規範

狀態（Status）：`PENDING`
日期（Date）：2026-06-27
範圍（Scope）：僅限文件語言與可讀性規則

## 目的

這份文件定義 AI Governance 文件的中文閱讀規則。目標不是消滅英文，
而是把英文收斂到必要位置，讓人類閱讀時以中文為主，工程追蹤時仍能
準確對上檔名、路徑、腳本、測試、變異 ID、欄位名稱與原始輸出。

核心原則：

```text
人看的句子用中文。
機器要對上的檔名、路徑、腳本、測試、變異 ID、欄位名稱與原始輸出保留原文。
保留原文的識別字，在正文中一律用 backtick 包起來。
```

## 狀態定義

用語規範文件使用下列狀態：

| 狀態 | 意義 |
| --- | --- |
| `PENDING` | 草案或候選規範 |
| `ADOPTED` | 已被某個範圍正式採用 |
| `SUPERSEDED` | 已被新版本取代 |

變異驗證文件使用下列狀態：

| 狀態 | 意義 |
| --- | --- |
| `PENDING` | 已建立但尚未完成驗證 |
| `VERIFIED` | 本地檢查已完成且符合預期 |
| `REVIEWED` | 人工審查已確認 |

## 適用範圍

適用於：

- AI Governance 說明文件；
- 實作包文件；
- 變異驗證文件；
- 人類閱讀用追蹤單；
- 知識地圖與入口文件；
- 下游複用儲存庫（consuming repo）採用 AI Governance 文件時的本地化整理。

不適用於：

- 程式碼識別字（identifier）；
- 檔名與路徑；
- 腳本檔名；
- 測試 ID；
- 變異 ID；
- 測試樣本檔名；
- CLI 原始輸出；
- log 原文；
- 結構描述欄位名稱（schema field name）。

這些不適用項目必須保留原文，並在正文中用 backtick 包起來。

## 固定用語表

| 英文 | 中文主稱 | 使用規則 |
| --- | --- | --- |
| mutation proof | 變異驗證 | 文件敘事使用中文；第一次可寫「變異驗證（mutation proof）」 |
| expected red | 預期紅燈 | 第一次可中英並列；後文用中文 |
| red check | 紅燈檢查 | 狀態回報用中文；腳本名保留英文 |
| survived | 變異逃過測試 | 不直接寫 survived，除非引用 CLI 原始輸出 |
| authority precedence | 權威優先序 | 概念用中文；測試或變異 ID 保留英文 |
| portable patch | 可攜補丁包 | 文件敘事用中文；資料夾或檔名保留英文 |
| ticket | 追蹤單 | 面向人類文件用中文 |
| fixture | 測試樣本 | 測試資料說明用中文；測試樣本檔名保留英文 |
| runner | 執行腳本 | 避免直接寫 runner；腳本檔名保留英文 |
| claim ceiling | 宣稱上限 | 治理語意用中文；審查回執欄位保留英文 |
| review receipt | 審查回執 | 文件敘事用中文；`REVIEW_RECEIPT` 保留英文 |
| push gate | 推送閘門 | 概念用中文；審查回執欄位保留英文 |
| canonical writer | 標準寫入器 | 記憶寫入敘事用中文；工具名保留英文 |
| repo | 儲存庫 | 一般敘事用中文；路徑或儲存庫名稱保留原文 |
| consuming repo | 下游複用儲存庫 | 第一次可中英並列；後文用中文 |
| repo-local | 儲存庫本地 | 第一次可寫「儲存庫本地（repo-local）」 |
| runtime | 執行環境 | 第一次可寫「執行環境（runtime）」 |
| memory protocol | 記憶協議 | 第一次可中英並列；檔名或工具名保留英文 |
| CLI output | CLI 原始輸出 | 原始內容保留英文並包 backtick |
| exit code | 結束碼 | 第一次可寫「結束碼（exit code）」 |
| schema field | 結構描述欄位 | 欄位名稱本身保留原文 |

## 第一次出現規則

英文概念第一次出現時，使用「中文主稱（英文）」。

範例：

```text
本文件定義變異驗證（mutation proof）的閱讀格式。
後文統一稱為變異驗證。
```

如果英文是識別字、檔名、路徑、欄位名稱或原始輸出，不翻譯，只包
backtick。

範例：

```text
對應變異 ID：`weaken_authority_precedence`
紅燈檢查腳本：`scripts/run_red_check.py`
```

## 改寫規則

| 原文傾向 | 建議寫法 |
| --- | --- |
| expected red 驗證 | 預期紅燈驗證 |
| mutation survived | 變異逃過測試 |
| portable patch | 可攜補丁包 |
| third ticket | 第三張追蹤單 |
| runner 執行 | 透過執行腳本執行 |
| runner failed | 執行腳本回報失敗 |
| red check fail | 紅燈檢查回報失敗 |
| tests fail | 測試失敗 |
| authority precedence 被 weaken | 權威優先序被弱化 |

## 變異驗證文件模板

變異驗證文件應使用固定結構，讓正文中文化，識別字集中管理。

```markdown
# AI Governance 變異驗證：<中文主題>

狀態：`PENDING | VERIFIED | REVIEWED`

## 目的

用中文描述這個錯誤變異要驗證的治理假設。

## 對應識別字

- 變異 ID：`<mutation_id>`
- 紅燈檢查腳本：`<script path>`
- 測試檔：`<test path>`
- 測試樣本：`<fixture path when needed>`

## 預期結果

套用錯誤變異後，對應測試必須失敗。

若測試沒有失敗，表示變異逃過測試，紅燈檢查執行腳本應以非零結束碼回報。

## 驗證紀錄

- 語法檢查：<通過 | 未執行>
- 測試結果：<預期失敗 | 未執行>
- 紅燈檢查腳本結果：<符合預期 | 不符合預期 | 未執行>
- 原始訊息：`<raw output when needed>`

## 宣稱上限

本文件只能宣稱這個變異案例是否被對應測試覆蓋。

不得宣稱整個治理模型正確。
不得宣稱其他變異案例已被覆蓋。
不得宣稱上游儲存庫已採用本文件。
```

## 掃描清單

整理文件時，逐段問四個問題：

1. 這個英文是程式碼、檔案、測試、變異 ID 或 CLI 原始輸出嗎？
2. 如果不是，是否已有中文主稱？
3. 如果是第一次出現，是否中英並列？
4. 後文是否改用中文主稱？

判斷方式：

| 狀況 | 處理 |
| --- | --- |
| `run_red_check.py` | 保留英文，包 backtick |
| `expected_red` | 保留英文，包 backtick |
| expected red 驗證 | 改成「預期紅燈驗證」 |
| mutation survived | 改成「變異逃過測試」 |
| portable patch | 改成「可攜補丁包」 |
| third ticket | 改成「第三張追蹤單」 |

## 推廣到其他儲存庫的方式

這份文件目前是本儲存庫內的候選用語規範。

其他儲存庫若要採用，應該：

1. 保留本文件的核心原則；
2. 只調整儲存庫本地詞彙；
3. 不翻譯程式碼識別字、腳本檔名、測試 ID、變異 ID、欄位名稱或原始輸出；
4. 不用本文件覆蓋該儲存庫自己的安全、權限或執行環境規則；
5. 把本地採用狀態標成 `PENDING`，直到該儲存庫完成文件審查。

本文件不授權跨儲存庫寫入。
本文件不自動成為任何下游或上游儲存庫的正式規範。

## 非目標

這份文件不做以下事情：

- 不修改程式碼；
- 不修改測試；
- 不修改變異 ID；
- 不修改執行環境行為；
- 不改變記憶協議；
- 不授權跨儲存庫寫入；
- 不取代儲存庫本地治理規則；
- 不要求一次批次改完所有文件。

## 待處置建議

建議第一個採用切片：

```text
以權威優先序相關的變異驗證文件作為樣板，
套用「中文敘事 + 英文識別字」格式。
```

建議樣板對象：

- 變異 ID：`weaken_authority_precedence`
- 文件類型：變異驗證文件
- 驗證重點：可讀性沒有退化、工程追蹤沒有斷裂、宣稱上限沒有擴張

完成後，再用審查回執確認是否可將本文件狀態從 `PENDING` 調整為
`ADOPTED`。
